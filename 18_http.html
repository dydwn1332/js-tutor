18장HTTP와 Forms

클라이언트에서 서버로의 각 요청에는 요청을 이해하는 데 필요한 모든 정보가 포함되어야 하며 서버에 저장된 컨텍스트를 이용할 수 없습니다.

Roy Fielding, 아키텍처 스타일 및 네트워크 기반 소프트웨어 아키텍처 설계

하이퍼 텍스트 전송 프로토콜(Hypertext Transfer Protocol)은 데이터를 요청하는 월드 와이드 웹에서 제공되는 메커니즘입니다. 이 페이지에서는 프로토콜에 대해 자세히 설명하고 브라우저 JavaScript가 프로토콜에 액세스하는 방법을 설명합니다.

Protocol

브라우저의 주소 표시 줄에 eloquentjavascript.net/18_http.html을 입력하면, 브라우저는 먼저 eloquentjavascript.net와 관련된 서버의 주소를 찾고 HTTP 트래픽에 관한 기본 포트 80에 그것의 TCP 연결을 열기 위한 시도를 하게 됩니다. 서버가 존재하고 연결이 승인되면, 브라우저는 다음과 같이 무엇인가를 보낼 수 있습니다.

GET /18_http.html HTTP/1.1
Host: eloquentjavascript.net
User-Agent: Your browser's name

서버는 동일한 연결을 통해 응답합니다.

HTTP/1.1 200 OK
Content-Length: 65585
Content-Type: text/html
Last-Modified: Mon, 08 Jan 2018 10:29:45 GMT

<!doctype html>
... the rest of the document

브라우저는 빈 라인 다음에 응답 부분을 취해 HTML 문서로 표시합니다.

클라이언트가 보낸 정보를 요청이라 하는데, 다음 라인으로 시작합니다.

GET /18_http.html HTTP/1.1

첫 번째 단어는 요청 method입니다. GET은 지정된 리소스를 취하고자 함을 의미합니다. 기타 일반적인 메서드로 DELETE는 리소스 삭제, PUT은 생성 또는 대체를 나타내고 POST는 정보를 전송하는데 사용됩니다. 서버는 받는 모든 요청을 수행할 의무는 없습니다. 임의의 웹사이트로 들어가 메인 페이지에 DELETE를 사용하면 아마 거절할 것입니다.

메서드 이름 뒤의 부분은 요청이 적용되는 리소스의 경로입니다. 가장 간단한 경우 리소스는 단순히 서버의 파일이지만 프로토콜에서는 필요하지 않습니다. 리소스는 파일처럼 전송할 수 있는 모든 것이 될 수 있습니다. 많은 서버가 즉석에서 생성하는 응답을 생성합니다. 예를 들어 https://github.com/marijnh를 열면 서버는 데이터베이스에서 "marijnh"라는 사용자를 찾고, 찾으면 해당 사용자에 대한 프로필 페이지를 생성합니다.

리소스 경로 다음에 있는 요청의 첫 번째 라인 HTTP/1.1은 사용 중인 HTTP 프로토콜의 버전을 나타냅니다.

대부분의 사이트에서는 HTTP 버전 2를 사용합니다. HTTP 버전 2는 버전 1.1과 동일한 개념을 지원하지만 훨씬 더 복잡하여 더 빠를 수 있습니다. 브라우저는 주어진 서버와 통신할 때 자동으로 적절한 프로토콜 버전으로 전환하며, 사용되는 버전에 관계없이 요청 결과는 동일합니다. 버전 1.1이 더 간단하고 다루기 쉽기 때문에 그것에 집중할 것입니다.

서버의 응답도 버전으로 시작하고 응답 상태가 먼저 세 자리 상태 코드이고, 그 다음에는 사람이 읽을 수 있는 문자열로 시작합니다.

HTTP/1.1 200 OK

2로 시작하는 상태 코드는 요청이 성공했음을 나타냅니다. 4로 시작하는 코드는 요청에 문제가 있음을 의미합니다. 404는 아마도 가장 유명한 HTTP 상태 코드일 것입니다. 이는 리소스를 찾을 수 없음을 의미합니다. 5로 시작하는 코드는 서버에서 오류가 발생했으며 요청에 책임이 없음을 의미합니다.

요청 또는 응답의 첫 번째 라인 뒤에는 헤더가 여러 개 올 수 있습니다. 이것들은 요청 또는 응답에 대한 추가 정보를 지정하는 양식의 name: value 라인입니다. 다음 헤더는 예 응답의 일부입니다.

Content-Length: 65585
Content-Type: text/html
Last-Modified: Thu, 04 Jan 2018 14:05:30 GMT

이것은 응답 문서의 크기와 타입을 알려줍니다. 이 경우 65,585바이트의 HTML 문서입니다. 또한 해당 문서가 마지막으로 수정된 시간을 알려줍니다.

대부분의 헤더들의 경우에 클라이언트와 서버는 요청 또는 응답에 헤더를 포함할지 여부를 자유롭게 결정할 수 있습니다. 그러나 몇 가지가 필요합니다. 예를 들어, 호스트 이름을 지정 하는 Host 헤더는 서버가 단일 IP 주소에서 여러 호스트 이름을 제공할 수 있기 때문에 요청에 포함되어야 합니다. 이 헤더가 없으면 서버는 클라이언트가 말하려는 호스트 이름을 알 수 없습니다.

헤더 뒤에는 요청과 응답 모두 전송되는 데이터가 포함된 본문이 오는 빈 줄을 포함할 수 있습니다. GET 및 DELETE 요청은 데이터 함께 보내지 않지만, PUT 및 POST요청은 데이터와 함께 보냅니다. 유사하게 오류 응답과 같은 일부 응답 타입에는 본문이 필요하지 않습니다.

Browsers와 HTTP

위 예에서 보았듯이 브라우저는 주소 표시줄에 URL을 입력하면 요청을 생성합니다. 결과 HTML 페이지가 이미지 및 JavaScript 파일과 같은 다른 파일을 참조하는 경우 해당 파일도 검색됩니다.

적당히 복잡한 웹사이트는 10~200개의 리소스를 쉽게 포함할 수 있습니다. 이를 빠르게 가져올 수 있도록 브라우저는 한 번에 하나씩 응답을 기다리지 않고 동시에 여러 요청을 수행합니다.

HTML 페이지에는 사용자가 정보를 입력하고 서버로 보낼 수 있는 forms가 포함될 수 있습니다. 다음은 양식의 예입니다.

<form method="GET" action="example/message.html">
  <p>Name: <input type="text" name="name"></p>
  <p>Message:<br><textarea name="message"></textarea></p>
  <p><button type="submit">Send</button></p>
</form>

이 코드는 두 개의 필드가 있는 양식을 설명합니다. 이름을 묻는 작은 필드와 메시지를 작성할 큰 필드입니다. 보내기 버튼을 클릭하면 양식이 제출됩니다 . 이는 해당 필드의 내용이 HTTP로 압축됨을 의미합니다. 요청하고 브라우저는 해당 요청의 결과로 이동합니다.

<form> 요소의 method 속성이 GET일 때(또는 누락되), 양식에 포함된 정보가 action URL의 끝에 query string로 추가되고 브라우저는 다음과 같이 URL에 요청을 생성합니다.

GET /example/message.html?name=Jean&message=Yes%3F HTTP/1.1

물음표는 URL의 경로 부분의 끝과 쿼리의 시작을 나타냅니다. 그 다음에 양식 필드 요소의 name 속성에 해당하는 이름과 값들의 쌍이 작성됩니다. 앰퍼샌드 문자(&)는 쌍을 구분하는 데 사용됩니다.

URL에 인코딩된 실제 메시지는 "예?"이지만 물음표는 이상한 코드로 대체됩니다. 쿼리 문자열의 일부 문자는 이스케이프 문자를 사용합니다. %3F로 표시된 물음표가 그 중 하나입니다. 모든 형식에는 이스케이프 문자를 사용하는 고유한 방법이 있습니다. URL encoding이라 하는 이 코드는 문자 코드를 인코딩하는 퍼센트 기호 다음에 두 개의 16진수(기본 16) 숫자를 사용합니다. 이 경우 3F는 십진법으로 63으로 물음표 문자의 코드입니다. 자바 스크립트는 이러한 형식을 인코딩 및 디코딩 하기 위해 encodeURIComponent와 decodeURIComponent 함수를 사용합니다.

console.log(encodeURIComponent("Yes?"));
// → Yes%3F
console.log(decodeURIComponent("Yes%3F"));
// → Yes?

앞 예에서 HTML 양식의 method 속성을 POST로 변경하면 HTTP 요청은 쿼리 문자열을 URL에 추가하는 대신 요청 본문에 넣습니다.

POST /example/message.html HTTP/1.1
Content-length: 24
Content-type: application/x-www-form-urlencoded

name=Jean&message=Yes%3F

GET 요청은 부작용이 없지만 단순히 정보를 요청하는 것에 사용해야 합니다. 예를 들어 새 계정을 만들거나 메시지를 게시하는 것과 같이 서버에서 무언가를 변경하는 요청은 POST와 같은 다른 방법으로 표현해야 합니다. 브라우저와 같은 클라이언트 측 소프트웨어는 맹목적으로 POST 요청을 해서는 안 되지만 종종 암시적으로 GET 요청을 합니다(사용자가 곧 필요로 할 것으로 생각되는 리소스를 미리 가져오기).

Fetch

브라우저 JavaScript가 HTTP 요청을 할 수 있는 인터페이스를 fetch라 합니다. 비교적 새롭기 때문에 Promise를 편리하게 사용합니다.

fetch("example/data.txt").then(response => {
  console.log(response.status);
  // → 200
  console.log(response.headers.get("Content-Type"));
  // → text/plain
});

fetch 호출은 서버 응답 관련 정보를 포함하는 Response 객체로 해결하는 promise를 반환합니다(상태 코드와 그것의 헤더와 같은). 헤더는 이름이 대소문자를 구분하지 않아야 하기 때문에 그것이 갖는 키들을 취급하기 위한 객체로 Map 구조를 사용합니다. 이는 headers.get("Content-Type")와 headers.get("content-TYPE")이 같은 값을 반환함을 의미합니다.

fetch에 의해 반환되는 약속은 서버가 에러 코드를 나타낼지라도 성공적으로 약속을 지킵니다. 네트워크 에러가 있거나 또는 요청 주소를 찾을 수 없으면 약속을 지킬 수 없습니다.

fetch에 대한 첫 번째 인수는 요청해야 하는 URL입니다. 해당 URL이 프로토콜 이름으로 시작하지 않는 경우에(http:와 같은) 그것은 relative로 처리됩니다. 그것은 현재 문서를 상대적으로 해석하는 것을 의미합니다. 슬래시(/)로 시작하면 서버 이름 뒤 부분인 현재 경로를 대체합니다. 그렇지 않으면 마지막 슬래시 문자를 포함하여 현재 경로의 일부가 상대적인 URL 앞에 놓입니다.

응답의 실제 내용을 얻으려면 해당 text 메서드를 사용할 수 있습니다. 초기 약속은 응답의 헤더가 수신되는 즉시 해결되고 응답 본문을 읽는 데 시간이 더 오래 걸릴 수 있으므로 다시 약속을 반환합니다.

fetch("example/data.txt")
  .then(resp => resp.text())
  .then(text => console.log(text));
// → This is the content of data.txt

json이라는 유사한 메서드는 본문을 JSON으로 구문 분석할 때 얻은 값으로 확인하거나 유효한 JSON이 아닌 경우 거부하는 약속을 반환합니다.

기본적으로 fetch는 GET 메서드를 사용하여 요청하고 요청 본문을 포함하지 않습니다. 추가 옵션이 있는 객체를 두 번째 인수로 전달하여 다르게 구성할 수 있습니다. 예를 들어 다음 요청은 example/data.txt을 삭제하려고 합니다.

fetch("example/data.txt", {method: "DELETE"}).then(resp => {
  console.log(resp.status);
  // → 405
});

상태 코드 405는 "허용되지 않는 방법"을 의미하며 HTTP 서버가 "나는 할 수 없습니다."라고 말하는 방식입니다.

요청 본문을 추가하기 위해 body 옵션을 포함할 수 있습니다. 헤더를 설정하는 headers 옵션이 있습니다. 예를 들어, 다음 요청에는 응답의 일부만 반환하도록 서버에 지시하는 Range 헤더가 포함되어 있습니다.

fetch("example/data.txt", {headers: {Range: "bytes=8-19"}})
  .then(resp => resp.text())
  .then(console.log);
// → the content

브라우저는 "호스트"와 같은 일부 요청 헤더와 서버가 본문의 크기를 파악하는 데 필요한 헤더를 자동으로 추가합니다. 그러나 고유한 헤더를 추가하면 인증 정보와 같은 항목을 포함하거나 서버에 수신하려는 파일 형식을 알려주는 데 유용합니다.

HTTP sandboxing

웹 페이지 스크립트에서 HTTP 요청을 하면 보안에 대한 우려가 다시 한 번 발생합니다. 스크립트를 제어하는 사람은 스크립트가 실행되는 컴퓨터에서 스크립트를 제어하는 사람과 관심사가 다를 수 있습니다. 보다 구체적으로 themafia.org를 방문하는 경우 브라우저의 식별 정보를 사용하여 모든 돈을 임의의 계정으로 이체하라는 지시 사항을 사용하여 해당 스크립트가 mybank.com에 요청할 수 있기를 원하지 않습니다.

이러한 이유로 브라우저는 스크립트가 다른 도메인(themafia.org 및 mybank.com과 같은 이름)에 HTTP 요청을 하는 것을 허용하지 않음으로써 보호합니다.

이것은 정당한 이유로 여러 도메인에 액세스하려는 시스템을 구축할 때 성가신 문제가 될 수 있습니다. 다행히도 서버는 응답에 다음과 같은 헤더를 포함하여 다른 도메인에서 오는 요청이 괜찮다는 것을 브라우저에 명시적으로 나타낼 수 있습니다.

Access-Control-Allow-Origin: *

HTTP 감사

브라우저(클라이언트 측)에서 실행되는 JavaScript 프로그램과 서버(서버 측)의 프로그램 간의 통신이 필요한 시스템을 구축할 때 이 통신을 모델링하는 여러 가지 방법이 있습니다.

일반적으로 사용되는 모델은 원격 프로시저 호출(remote procedure calls) 모델입니다. 이 모델에서 통신은 함수가 실제로 다른 시스템에서 실행되고 있다는 점을 제외하고는 일반적인 함수 호출 패턴을 따릅니다. 호출에는 함수의 이름과 인수가 포함된 서버에 대한 요청이 포함됩니다. 해당 요청에 대한 응답에는 반환된 값이 포함됩니다.

원격 프로시저 호출의 관점에서 생각할 때 HTTP는 통신 수단일 뿐이며 이를 완전히 숨기는 추상화 계층을 작성할 가능성이 큽니다.

또 다른 접근 방식은 리소스 및 HTTP 메서드의 개념을 중심으로 통신을 구축하는 것입니다. addUser라는 원격 프로시저 대신에 /users/larry에 대한 PUT 요청을 사용합니다. 함수 인수에서 해당 사용자의 속성을 인코딩하는 대신 사용자를 나타내는 JSON 문서 형식을 정의하거나 기존 형식을 사용합니다. 새 리소스를 생성하라는 PUT 요청의 본문은 그런 문서입니다. 리소스는 리소스를 나타내는 문서를 다시 반환하는 리소스의 URL(예: /user/larry)을 GET 요청하여 가져옵니다.

이러한 두 번째 접근 방식을 사용하면 리소스 캐싱 지원(빠른 액세스를 위해 클라이언트에 복사본 유지)과 같이 HTTP가 제공하는 일부 기능을 더 쉽게 사용할 수 있습니다. 잘 설계된 HTTP에서 사용되는 개념은 서버 인터페이스를 설계하는데 유용한 원칙 집합을 제공할 수 있습니다.

Security 및 HTTPS

인터넷을 통해 이동하는 데이터는 길고 위험한 길을 따라가는 경향이 있습니다. 목적지에 도달하려면 커피숍 Wi-Fi 핫스팟에서 다양한 회사 및 공공에서 관리하는 네트워크에 이르기까지 모든 것을 통과해야 합니다. 경로를 따라 어느 지점에서나 검사하거나 수정할 수 있습니다.

이메일 계정의 비밀번호와 같이 비밀로 유지되는 것이 중요하거나 은행 웹사이트를 통해 송금하는 계좌 번호와 같이 수정되지 않은 상태로 목적지에 도착하는 것이 중요한 경우 일반 HTTP로는 충분하지 않습니다.

https://로 시작하는 URL에 사용되는 보안 HTTP 프로토콜은 HTTP 트래픽을 읽고 조작하기 어렵게 만드는 방식으로 래핑합니다. 데이터를 교환하기 전에 클라이언트는 브라우저가 인식하는 인증 기관에서 발급한 암호화 인증서가 있음을 증명하도록 요청하여 서버가 자신이 주장하는 사람인지 확인합니다. 다음으로, 연결을 통해 이동하는 모든 데이터는 도청 및 변조를 방지해야 하는 방식으로 암호화됩니다.

따라서 HTTPS가 제대로 작동할 때 HTTPS는 다른 사람들이 귀하가 대화하려는 웹사이트를 가장하거나 귀하의 통신을 스누핑하는 것을 방지합니다. 완벽하지 않고 인증서 위조나 도난, 소프트웨어 파손 등으로 HTTPS가 실패하는 사건이 여러 차례 있었지만 일반 HTTP보다 훨씬 안전합니다.

Form 필드

Form은 원래 웹 사이트에서 HTTP 요청으로 사용자가 제출한 정보를 보낼 수 있도록 사전 JavaScript 웹용으로 설계되었습니다. 이 디자인은 서버와의 상호 작용이 항상 새 페이지로 이동하여 발생한다고 가정합니다.

그러나 해당 요소는 페이지의 나머지 부분과 마찬가지로 DOM의 일부이며 양식 필드를 나타내는 DOM 요소는 다른 요소에 없는 여러 속성과 이벤트를 지원합니다. 이를 통해 JavaScript 프로그램으로 이러한 입력 필드를 검사 및 제어하고 양식에 새 기능을 추가하거나 양식 및 필드를 JavaScript 응용 프로그램의 빌딩 블록으로 사용하는 등의 작업을 수행할 수 있습니다.

웹 양식은 <form> 태그로 그룹화된 입력 필드 수로 구성됩니다. HTML은 간단한 켜기/끄기 확인란에서 드롭다운 메뉴 및 텍스트 입력 필드에 이르기까지 다양한 스타일의 필드를 허용합니다. 여기에서는 모든 필드 타입을 포괄적으로 논의하지는 않겠지만 대략적인 개요부터 시작합니다.

많은 필드 유형이 <input> 태그를 사용합니다 . 이 태그의 type 속성은 필드의 스타일을 선택하는 데 사용됩니다. 다음은 일반적으로 사용되는 몇 가지 <input> 타입입니다.

text단일 라인 텍스트 필드
passwordtext와 같지만 텍스트를 은닉함.
checkboxon/off 스위치
radio다중 선택 필드 부분
file사용자가 파일을 선택할 수 있음.

양식 필드가 반드시 <form> 태그에 나타날 필요는 없습니다. 페이지의 아무 곳에나 넣을 수 있습니다. 이러한 형식이 없는 필드는 제출할 수 없지만(전체 양식만 가능) JavaScript로 입력에 응답할 때 어쨌든 일반적으로 필드를 제출하고 싶지 않은 경우가 많습니다.

<p><input type="text" value="abc"> (text)</p>
<p><input type="password" value="abc"> (password)</p>
<p><input type="checkbox" checked> (checkbox)</p>
<p><input type="radio" value="A" name="choice">
   <input type="radio" value="B" name="choice" checked>
   <input type="radio" value="C" name="choice"> (radio)</p>
<p><input type="file"> (file)</p>

이러한 요소에 대한 JavaScript 인터페이스는 요소 타입에 따라 다릅니다.

여러 줄 텍스트 필드에는 고유한 태그 <textarea>가 있습니다. 대부분 속성을 사용하여 여러 줄 시작 값을 지정하는 것이 어색하기 때문입니다. <textarea> 태그는 태그를 닫는 </textarea> 대응이 필요하고, 시작 텍스트로 value 속성 대신에 사이에 텍스트를 사용합니다.

<textarea>
one
two
three
</textarea>

마지막으로 <select> 태그는 사용자가 미리 정의된 여러 옵션 중에서 선택할 수 있는 필드를 만드는 데 사용됩니다.

<select>
  <option>Pancakes</option>
  <option>Pudding</option>
  <option>Ice cream</option>
</select>

양식 필드의 값이 변경될 때마다 "change"이벤트가 발생합니다.

Focus

HTML 문서의 대부분의 요소와 달리 양식 필드는 keyboard focus를 얻을 수 있습니다. 클릭하거나 다른 방식으로 활성화하면 현재 활성 요소가 되고 키보드 입력을 받습니다.

따라서 텍스트 필드에 포커스가 있을 때만 입력할 수 있습니다. 다른 필드는 키보드 이벤트에 다르게 응답합니다. 예를 들어 <select> 메뉴는 사용자가 입력한 텍스트가 포함된 옵션으로 이동하려고 시도하고 선택 항목을 위아래로 이동하여 화살표 키에 응답합니다.

focus 및 blur 메서드를 사용하여 JavaScript에서 포커스를 제어할 수 있습니다. 첫 번째는 호출된 DOM 요소로 포커스를 이동하고 두 번째는 포커스를 제거합니다. document.activeElement의 값은 현재 포커스가 있는 요소에 해당합니다.

<input type="text">
<script>
  document.querySelector("input").focus();
  console.log(document.activeElement.tagName);
  // → INPUT
  document.querySelector("input").blur();
  console.log(document.activeElement.tagName);
  // → BODY
</script>

일부 페이지의 경우 사용자는 양식 필드와 즉시 상호 작용하기를 원할 것으로 예상됩니다. JavaScript는 문서가 로드될 때 이 필드에 초점을 맞추는 데 사용할 수 있지만 HTML은 autofocus속성도 제공합니다. 이 속성은 동일한 효과를 생성하는 동시에 브라우저에 우리가 달성하려는 것을 알려줍니다. 이렇게 하면 사용자가 다른 것에 초점을 맞춘 경우와 같이 적절하지 않은 경우 동작을 비활성화할 수 있는 옵션이 브라우저에 제공됩니다.

브라우저에서는 전통적으로 사용자가 tab 키를 눌러 문서 전체에서 포커스를 이동할 수도 있습니다. tabindex 속성으로 요소가 포커스를 받는 순서에 영향을 줄 수 있습니다. 다음 예에서는 먼저 도움말 링크를 거치지 않고 텍스트 입력에서 확인 버튼으로 포커스를 이동합니다.

<input type="text" tabindex=1> <a href=".">(help)</a>
<button onclick="console.log('ok')" tabindex=2>OK</button>

기본적으로 대부분의 HTML 요소 타입은 focus를 사용할 수 없습니다. 그러나 tabindex 속성을 추가하여 모든 요소에 focus를 줄 수 있습니다. tabindex가 -1이면 일반적으로 포커스를 받을 수 있는 경우에도 탭을 사용하여 요소를 건너뜁니다.

필드 작동 중지

모든 양식 필드는 해당 disabled 속성을 통해 비활성화 할 수 있습니다. 값이 없이 지정할 수 있는 속성입니다.

<button>I'm all right</button>
<button disabled>I'm out</button>

비활성화된 필드는 focus 또는 변경할 수 없으며 브라우저는 이를 회색으로 보이게 합니다.

프로그램이 서버와의 통신이 필요할 수 있는 일부 버튼 또는 기타 컨트롤로 인해 발생하는 작업을 처리하는 중이어서 시간이 오래 걸리는 경우 작업이 완료될 때까지 컨트롤을 비활성화하는 것이 좋습니다. 그렇게 하면 사용자가 조바심내고 다시 클릭할 때 실수로 작업을 반복하지 않습니다.

전체로서 양식

필드가 <form> 요소에 포함되면, 그것의 DOM 요소는 양식의 DOM 요소를 다시 링크하는 form 속성을 가집니다. <form> 요소는 배열을 포함하는 elements라는 속성을 가집니다.

양식 필드의 name 속성은 양식이 제출될 때 값이 식별되는 방식을 결정합니다. 양식의 elements 속성에 접근될 때 속성 이름으로 사용될 수 있는데, 그것은 수에 의해 접근되는 객체와 같은 배열 및 이름으로 접근 가능한 맵처럼 작동합니다.

<form action="example/submit.html">
  Name: <input type="text" name="name"><br>
  Password: <input type="password" name="password"><br>
  <button type="submit">Log in</button>
</form>
<script>
  let form = document.querySelector("form");
  console.log(form.elements[1].type);
  // → password
  console.log(form.elements.password.type);
  // → password
  console.log(form.elements.name.form == form);
  // → true
</script>

submit의 type 속성을 갖는 버튼은 그것이 사용될 때 양식이 제출되도록 합니다. 양식 필드가 포커스 상태일 때 enter를 누르면 같은 효과가 발생합니다.

일반적으로 양식을 제출한다는 것은 브라우저가 GET 또는 POST 요청을 사용하여 양식의 action 속성이 나타내는 페이지를 탐색한다는 것을 의미합니다. 하지만 그 전에 "submit" 이벤트가 발생합니다. JavaScript로 이 이벤트를 처리하고 이벤트 객체 상에서 preventDefault를 호출하여 이 기본 동작을 방지할 수 있습니다.

<form action="example/submit.html">
  Value: <input type="text" name="value">
  <button type="submit">Save</button>
</form>
<script>
  let form = document.querySelector("form");
  form.addEventListener("submit", event => {
    console.log("Saving value", form.elements.value.value);
    event.preventDefault();
  });
</script>

JavaScript에서 "submit" 이벤트를 가로채는 것은 다양한 용도로 사용됩니다. 사용자가 입력한 값이 의미가 있는지 확인하고 양식을 제출하는 대신 오류 메시지를 즉시 표시하는 코드를 작성할 수 있습니다. 또는 예에서와 같이 양식을 완전히 제출하는 일반적인 방법을 비활성화하고 프로그램이 입력을 처리하도록 할 수 있습니다. 페이지를 다시 로드하지 않고 서버로 보내는 데 fetch를 사용할 수 있습니다.

텍스트 필드

<textarea> 태그로 생성된 필드 또는 text 또는 password의 타입을 갖는 <input> 태그는 공통 인터페이스를 공유합니다. DOM 요소에는 현재 내용을 문자열 값으로 보유하는 value 속성이 있습니다. 이 속성을 다른 문자열로 설정하면 필드의 내용이 변경됩니다.

텍스트 필드의 selectionStart 및 selectionEnd 속성은 텍스트의 커서 및 선택 항목에 대한 정보를 제공합니다. 아무것도 선택하지 않은 경우 이 두 속성은 커서 위치를 나타내는 동일한 번호를 보유합니다. 예를 들어 0은 텍스트의 시작을 나타내고 10은 커서가 10번째 문자 뒤에 있음을 나타냅니다. 필드의 일부가 선택되면 두 속성이 달라서 선택한 텍스트의 시작과 끝을 알려줍니다. value와 같이 이러한 속성도 쓸 수 있습니다.

Khasekhemwy에 대한 기사를 쓰고 있지만 그의 이름을 철자하는데 문제가 있다고 상상해보십시오. 다음 코드는 F2 키를 누를 때 문자열 "Khasekhemwy"를 삽입하는 이벤트 처리기와 <textarea> 태그를 연결합니다.

<textarea></textarea>
<script>
  let textarea = document.querySelector("textarea");
  textarea.addEventListener("keydown", event => {
    // The key code for F2 happens to be 113
    if (event.keyCode == 113) {
      replaceSelection(textarea, "Khasekhemwy");
      event.preventDefault();
    }
  });
  function replaceSelection(field, word) {
    let from = field.selectionStart, to = field.selectionEnd;
    field.value = field.value.slice(0, from) + word +
                  field.value.slice(to);
    // Put the cursor after the word
    field.selectionStart = from + word.length;
    field.selectionEnd = from + word.length;
  }
</script>

replaceSelection 함수는 텍스트 필드 내용의 현재 선택된 부분을 주어진 단어로 대체한 다음 사용자가 계속 입력할 수 있도록 해당 단어 뒤에 커서를 이동합니다.

텍스트 필드에 관한 "change" 이벤트는 무엇인가 입력될 때마다 작동되는 것은 아니고 콘텐츠가 변경된 후에 필드가 포커스를 상실할 때 작동됩니다. 텍스트 필드에 대한 변경에 즉시 응답하기 위해서는 "input" 이벤트에 관한 처리기를 등록해야 합니다. 그것은 사용자가 문자를 입력하거나 텍스트를 삭제하거나 또는 필드의 콘텐츠를 변경할 때 작동됩니다.

다음 예는 필드에 테스트의 현재 길이를 전시하는 텍스트 필드와 카운터를 나타냅니다.

<input type="text"> length: <span id="length">0</span>
<script>
  let text = document.querySelector("input");
  let output = document.querySelector("#length");
  text.addEventListener("input", () => {
    output.textContent = text.value.length;
  });
</script>

체크 상자와 라디오 버튼

체크 상자 필드는 바이너리 토글입니다. 부울 값을 보유하는 속성을 통해 값을 추출하거나 변경할 수 있습니다.

<label>
  <input type="checkbox" id="purple"> Make this page purple
</label>
<script>
  let checkbox = document.querySelector("#purple");
  checkbox.addEventListener("change", () => {
    document.body.style.background =
      checkbox.checked ? "mediumpurple" : "";
  });
</script>

<label> 태그는 입력 필드를 문서의 일부와 연관시킵니다. 라벨의 아무 곳이나 클릭하면 필드가 활성화되어 필드에 초점을 맞추고 확인란 또는 라디오 버튼일 때 값을 토글합니다.

라디오 버튼은 체크박스와 비슷하지만 같은 name 속성을 가진 다른 라디오 버튼과 암시적으로 연결되어 있어 한 번에 하나만 활성화 할 수 있습니다.

Color:
<label>
  <input type="radio" name="color" value="orange"> Orange
</label>
<label>
  <input type="radio" name="color" value="lightgreen"> Green
</label>
<label>
  <input type="radio" name="color" value="lightblue"> Blue
</label>
<script>
  let buttons = document.querySelectorAll("[name=color]");
  for (let button of Array.from(buttons)) {
    button.addEventListener("change", () => {
      document.body.style.background = button.value;
    });
  }
</script>

querySelectorAll로 주어진 CSS 쿼리에서 대괄호는 속성 부합에 사용됩니다. name 속성이 "color"인 요소를 선택합니다.

필드 선택

Select 필드는 개념적으로 라디오 버튼과 유사하며 사용자가 옵션 집합에서 선택할 수도 있습니다. 그러나 라디오 버튼이 옵션 레이아웃을 제어하는 경우 <select> 태그의 모양은 브라우저에 의해 결정됩니다.

선택 필드에는 라디오 상자가 아닌 확인란 목록과 더 유사한 변형이 있습니다. multiple 속성이 지정되는 경우에 <select> 태그를 사용하면 사용자가 단일 옵션이 아니라 여러 옵션을 선택할 수 있습니다. 이것은 대부분의 브라우저에서 일반 선택 필드와 다르게 표시되며 일반적으로 열 때만 옵션을 표시 하는 드롭다운 컨트롤 로 그려집니다.

각 <option> 태그는 값을 가집니다. 이 값은 value 속성으로 정의할 수 있습니다. 이것이 주어지지 않으면 옵션 안의 텍스트가 값으로 계산됩니다. <select> 요소의 value 속성은 현재 선택한 옵션을 반영합니다. multiple 필드는 많은 것을 의미하는 것이 아니고 현재 선택한 옵션들 중에서 오직 하나의 값만을 제공합니다.

<select> 필드의 <option> 태그는 배열로 접근될 수 있습니다. 각 옵션은 selected라는 속성을 갖는데 이것은 해당 옵션이 현재 선택되어 있는지 여부를 나타냅니다. 옵션 선택 및 해제를 위해 속성을 사용할 수 있습니다.

다음 예는 multiple 선택 필드에서 선택한 값을 추출하고 이를 사용하여 개별 비트에서 이진수를 구성합니다. 다중 옵션을 선택하기 위해 control 키를 사용합니다(또는 Mac의 경우 command).

<select multiple>
  <option value="1">0001</option>
  <option value="2">0010</option>
  <option value="4">0100</option>
  <option value="8">1000</option>
</select> = <span id="output">0</span>
<script>
  let select = document.querySelector("select");
  let output = document.querySelector("#output");
  select.addEventListener("change", () => {
    let number = 0;
    for (let option of Array.from(select.options)) {
      if (option.selected) {
        number += Number(option.value);
      }
    }
    output.textContent = number;
  });
</script>

File 필드

File 필드는 원래 양식을 통해 사용자의 컴퓨터에서 파일을 업로드하는 방법으로 설계되었습니다. 최신 브라우저에서는 JavaScript 프로그램에서 이러한 파일을 읽는 방법도 제공합니다. 필드는 일종의 게이트 키퍼 역할을 합니다. 스크립트는 단순히 사용자의 컴퓨터에서 개인 파일 읽기를 시작할 수 없지만 사용자가 이러한 필드에서 파일을 선택하면 브라우저는 해당 작업을 스크립트가 파일을 읽을 수 있음을 의미하는 것으로 해석합니다.

파일 필드는 일반적으로 옆에 선택한 파일에 대한 정보와 함께 "파일 선택" 또는 "찾아보기"와 같은 라벨이 붙은 버튼처럼 보입니다.

<input type="file">
<script>
  let input = document.querySelector("input");
  input.addEventListener("change", () => {
    if (input.files.length > 0) {
      let file = input.files[0];
      console.log("You chose", file.name);
      if (file.type) console.log("It has type", file.type);
    }
  });
</script>

파일 필드 요소의 files 속성은 필드에서 선택되는 파일들을 포함하는 객체와 같은 (실제 배열은 아님) 배열입니다. 초기에는 비어 있습니다. 단순 file 속성이 아닌 이유는 파일 필드들이 multiple 속성을 지원하기 때문인데, 그것은 동시에 여러 파일들을 선택할 수 있도록 합니다.

files 객체에서 객체들은 name (파일명), size (바이트 단위의 파일 크기) 및 type (text/plain 또는 image/jpeg과 같은 파일의 매체 타입)과 같은 속성들을 가집니다.

파일에 없는 것은 파일의 내용을 포함하는 속성입니다. 그것에 도달하는 것은 조금 더 복잡합니다. 디스크에서 파일을 읽는 데 시간이 걸릴 수 있으므로 문서가 정지되는 것을 방지하려면 인터페이스가 비동기식이어야 합니다.

<input type="file" multiple>
<script>
  let input = document.querySelector("input");
  input.addEventListener("change", () => {
    for (let file of Array.from(input.files)) {
      let reader = new FileReader();
      reader.addEventListener("load", () => {
        console.log("File", file.name, "starts with",
                    reader.result.slice(0, 20));
      });
      reader.readAsText(file);
    }
  });
</script>

파일 읽기는 FileReader 객체를 생성하고 객체에 대한 "load" 이벤트 핸들러를 등록하고 해당 readAsText 메서드를 호출하여 읽고자 하는 파일을 제공함으로써 수행됩니다. 로드가 완료되면 판독기의 result 속성에 파일의 내용이 포함됩니다.

FileReader들은 또한 파일 읽기에 실패할 때 "error" 이벤트를 작동시킵니다. 에러 객체 자체는 reader의 error 속성으로 종료됩니다. 이 인터페이스는 promises가 언어의 일부가 되기 전에 설계되었습니다. 다음과 같이 약속으로 래핑할 수 있습니다.

function readFileText(file) {
  return new Promise((resolve, reject) => {
    let reader = new FileReader();
    reader.addEventListener(
      "load", () => resolve(reader.result));
    reader.addEventListener(
      "error", () => reject(reader.error));
    reader.readAsText(file);
  });
}

클라이언트 측면의 데이터 저장

약간의 JavaScript가 포함된 간단한 HTML 페이지는 기본 작업을 자동화하는 작은 도우미 프로그램인 "미니 애플리케이션"에 적합한 형식이 될 수 있습니다. 몇 가지 양식 필드를 이벤트 처리기와 연결하면 센티미터와 인치 사이를 변환하는 것부터 마스터 암호와 웹사이트 이름에서 암호를 계산하는 것까지 무엇이든 할 수 있습니다.

이러한 애플리케이션이 섹션 사이에 무언가를 기억해야 하는 경우 JavaScript 바인딩을 사용할 수 없습니다. 이러한 바인딩은 페이지가 닫힐 때마다 버려집니다. 서버를 설정하고 인터넷에 연결하고 응용 프로그램이 거기에 무언가를 저장하도록 할 수 있습니다. 추후 20장에서 그렇게 하는 방법을 볼 것 입니다. 그러나 그것은 많은 추가 작업과 복잡성입니다. 때로는 브라우저에 데이터를 보관하는 것만으로도 충분합니다.

localStorage 객체는 페이지 재 로드를 위한 방식으로 데이터를 저장하는 데 사용할 수 있습니다. 이 객체를 사용하면 이름 아래에 문자열 값을 보관할 수 있습니다.

localStorage.setItem("username", "marijn");
console.log(localStorage.getItem("username"));
// → marijn
localStorage.removeItem("username");

localStorage의 값은 덮어 쓰거나, removeItem로 제거하거나 또는 사용자가 로컬 데이터를 비울 때까지 남아 있습니다.

다른 도메인의 사이트에는 다른 저장 구획이 있습니다. 즉 주어진 웹사이트에 localStorage로 저장된 데이터는 원칙적으로 동일한 사이트의 스크립트에서만 읽고 덮어쓸 수 있습니다.

브라우저는 사이트가 localStorage에 저장할 수 있는 데이터의 크기를 제한합니다. 사람들의 하드 드라이브를 정크로 채우는 것이 실제로 수익성이 없다는 사실과 함께 이러한 제한은 기능이 너무 많은 공간을 차지하는 것을 방지합니다.

다음 코드는 조잡한 메모 작성 응용 프로그램을 구현합니다. 명명된 메모 세트를 유지하고 사용자가 메모를 편집하고 새 메모를 만들 수 있습니다.

Notes: <select></select> <button>Add</button><br>
<textarea style="width: 100%"></textarea>

<script>
  let list = document.querySelector("select");
  let note = document.querySelector("textarea");

  let state;
  function setState(newState) {
    list.textContent = "";
    for (let name of Object.keys(newState.notes)) {
      let option = document.createElement("option");
      option.textContent = name;
      if (newState.selected == name) option.selected = true;
      list.appendChild(option);
    }
    note.value = newState.notes[newState.selected];

    localStorage.setItem("Notes", JSON.stringify(newState));
    state = newState;
  }
  setState(JSON.parse(localStorage.getItem("Notes")) || {
    notes: {"shopping list": "Carrots\nRaisins"},
    selected: "shopping list"
  });

  list.addEventListener("change", () => {
    setState({notes: state.notes, selected: list.value});
  });
  note.addEventListener("change", () => {
    setState({
      notes: Object.assign({}, state.notes,
                           {[state.selected]: note.value}),
      selected: state.selected
    });
  });
  document.querySelector("button")
    .addEventListener("click", () => {
      let name = prompt("Note name");
      if (name) setState({
        notes: Object.assign({}, state.notes, {[name]: ""}),
        selected: name
      });
    });
</script>

스크립트는 localStorage에 저장된 "Notes" 값에서 시작 상태를 가져오거나 누락된 경우 쇼핑 목록만 있는 예제 상태를 만듭니다. localStorage에 존재하지 않는 필드 읽기는 null입니다. null을 JSON.parse에 전달하면 문자열 "null"을 구문 분석하여 null을 반환합니다. 따라서 || 연산자는 이러한 상황에서 기본값을 제공하는데 사용됩니다.

setState 메서드는 DOM이 주어진 상태를 표시하고 있는지 확인하고 새 상태를 localStorage에 저장합니다. 이벤트 처리기는 이 함수를 호출하여 새 상태로 이동합니다.

예에서 Object.assign의 사용은 과거 state.notes를 복사하는 새로운 객체를 생성하는데 있지만, 하나의 속성이 추가되거나 덮어쓰여진 새 객체를 생성하기 위한 것 입니다. Object.assign은 첫 번째 인수를 취하고 추가 인수의 모든 속성을 여기에 추가합니다. 따라서 빈 객체를 제공하면 새 객체가 채워집니다. 세 번째 인수의 대괄호 표기법은 이름이 일부 동적 값을 기반으로 하는 속성을 만드는 데 사용됩니다.

sessionStorage라는 localStorage와 유사한 또 다른 객체가 있습니다. 둘의 차이점은 각 세션이 끝날때 sessionStorage의 내용이 잊혀진다는 것입니다. 이는 대부분의 브라우저에서 브라우저가 닫힐 때를 의미합니다.

요약

이 페이지에서는 HTTP 프로토콜이 어떻게 작동하는지 논의했습니다. client는 메서드(보통 GET)와 소스를 식별하는 경로를 포함하여 요청을 보냅니다. 서버는 다음 상태 코드와 응답 본문과 요청 및 응답 수행할 작업을 결정합니다. 요청과 응답 모두 추가 정보를 제공하는 헤더를 포함할 수 있습니다.

브라우저 JavaScript가 HTTP 요청을 할 수 있는 인터페이스를 fetch라 합니다. 요청은 다음과 같습니다.

fetch("/18_http.html").then(r => r.text()).then(text => {
  console.log(`The page starts with ${text.slice(0, 15)}`);
});

브라우저는 웹 페이지를 표시하는 데 필요한 리소스를 가져오도록 GET 요청을 합니다. 페이지에는 양식이 제출될 때 사용자가 입력한 정보를 새 페이지에 대한 요청으로 보낼 수 있는 양식도 포함될 수 있습니다.

HTML은 텍스트 필드, 확인란, 객관식 필드 및 파일 선택기와 같은 다양한 유형의 양식 필드를 나타낼 수 있습니다.

이러한 필드는 JavaScript로 검사하고 조작할 수 있습니다. 변경될 때 "change" 이벤트를 발생시키고, 텍스트가 입력될 때 "input" 이벤트를 발생시키고, 키보드 포커스가 있을 때 키보드 이벤트를 수신합니다. value (텍스트 및 선택 필드의 경우) 또는 checked (체크박스 및 라디오 버튼의 경우)와 같은 속성은 필드의 내용을 읽거나 설정하는 데 사용됩니다.

양식이 제출되면 "submit" 이벤트가 발생합니다. 해당 이벤트 상에서 브라우저의 기본 행위를 작동시키지 않기 위해서 preventDefault를 호출할 수 있습니다. 양식 필드 요소는 양식 태그 외부에서도 발생할 수 있습니다.

사용자가 파일 선택기 필드에서 로컬 파일 시스템의 파일을 선택하면 FileReader 인터페이스를 사용하여 JavaScript 프로그램에서 이 파일의 내용에 액세스할 수 있습니다.

localStorage와 sessionStorage 객체들은 페이지 재로드 방식으로 정보를 저장하는 데 사용할 수 있습니다. 첫 번째 객체는 데이터를 영원히(또는 사용자가 지우기로 결정할 때까지) 저장하고 두 번째 객체는 브라우저가 닫힐 때까지 데이터를 저장합니다.

연습 문제

콘텐츠 협상

HTTP가 할 수 있는 일 중 하나는 content negotiation입니다. Accept 요청 헤더는 클라이언트가 취하고자 하는 문서 타입이 무엇인지 말하는데 사용됩니다. 많은 서버가 이 헤더를 무시하지만 서버가 리소스를 인코딩하는 다양한 방법을 알고 있는 경우 이 헤더를 보고 클라이언트가 선호하는 헤더를 보낼 수 있습니다.

URL https://eloquentjavascript.net/author는 클라이언트가 요구하는 것에 따라 plaintext, HTML 또는 JSON로 응답하도록 구성됩니다. 이러한 형식은 표준화된 media types text/plain, text/html 및 application/json로 식별됩니다.

현재 리소스의 모든 세 가지 형식을 가져오도록 하는 요청을 보냅니다. 원하는 매체 타입으로 Accept라는 헤더를 설정하기 위해 fetch에 전달되는 옵션 객체에서 headers 속성을 사용하도록 합니다.

document.querySelector 또는 document.getElementById를 사용하여 HTML에 정의된 요소에 접근할 수 있도록 합니다. 버튼 상에서 "click" 또는 "mousedown" 이벤트에 관한 처리기는 텍스트 필드의 value 속성을 취하고 Function을 호출할 수 있습니다.

Function에 대한 호출과 try 블록에서 그에 관한 결과 호출 모두 가능하도록 해서 예외를 포착할 수 있도록 합니다. 이 경우에 어떤 예외 타입인지 모르기 때문에 모든 것을 포팍하도록 합니다.

출력 요소의 textContent 속성은 문자열 메시지로 채우는데 사용될 수 있습니다. 또는 과거 콘텐츠를 유지하고자 하면 document.createTextNode를 사용하여 새 텍스트 노드를 생성해서 요소에 그것을 붙입니다. 모든 출력이 한 줄에 나타나지 않도록 끝에 줄 바꿈 문자를 추가하는 것을 잊지 마십시오.

Conway의 생명 게임

Conway의 생명 게임은 격자에 인공 생명을 생성하는 간단한 시뮬레이션으로 각 세포는 살아있거나 그렇지 않습니다. 각 세대(턴)에는 다음 규칙이 적용됩니다.

  • 살아있는 이웃이 2개 미만이거나 3개 이상인 살아있는 세포는 다음 세대에는 죽습니다.

  • 2~3개의 살아있는 이웃이 있는 살아있는 세포는 다음 세대까지 살아갑니다.

  • 정확히 세 개의 살아있는 이웃이 있는 죽은 세포는 살아있는 세포가 됩니다.

neighbor는 인접하는 셀로 대각선으로 인접한 셀들을 포함합니다.

이러한 규칙은 한 번에 하나의 사각형이 아니라 전체 그리드에 한 번에 적용됩니다. 즉, 인접 셀의 계산은 생성 시작 시의 상황을 기반으로 하며, 이 생성 동안 인접 셀에 발생하는 변경 사항이 주어진 셀의 새로운 상태에 영향을 주어서는 안 됩니다.

적절한 데이터 구조를 사용하여 이 게임을 구현하십시오. 처음에 임의의 패턴으로 그리드를 채우는 데 Math.random을 사용합니다. 다음 세대로 넘어가기 위해 옆에 버튼이 있는 체크박스 필드 그리드로 표시합니다. 사용자가 체크박스를 선택하거나 선택을 취소할 때 다음 세대를 계산할 때 변경 사항을 포함해야 합니다.

변경 사항이 개념적으로 동시에 발생하는 문제를 해결하려면 하나의 그리드를 사용하고 다음 차례를 나타내는 새 그리드를 생성하는 순수 함수로 세대 계산을 나타냅니다.

행렬의 표현은 6장과 같은 방법으로 할 수 있다. 두 차원의 인접한 좌표를 반복하는 두 개의 중첩 루프로 라이브 이웃을 계산할 수 있습니다. 필드 외부의 셀을 세지 않도록 주의하고 우리가 셀 이웃의 중심에 있는 셀을 무시하십시오.

체크박스에 대한 변경 사항이 다음 세대에 적용되도록 하는 것은 두 가지 방법으로 수행할 수 있습니다. 이벤트 처리기는 이러한 변경 사항을 감지하고 이를 반영하도록 현재 그리드를 업데이트하거나 다음 차례를 계산하기 전에 체크박스의 값에서 새로운 그리드를 생성할 수 있습니다.

이벤트 처리기를 선택한 경우 변경할 셀을 쉽게 찾을 수 있도록 각 확인란이 해당하는 위치를 식별하는 속성을 첨부할 수 있습니다.

체크박스 격자를 그리려면 <table> 요소를 사용하거나 단순히 모든 요소를 동일한 요소에 넣고 행 사이에 <br>요소를 넣습니다.
